# 强制规则（AI 行为约束，优先级仅次于流程门禁）

> 本文件规定 AI 的**行为约束**（交互、Git、宏、路径、迭代上限）。
> 流程的**顺序与门禁纪律**（必须按步骤顺序执行、不可跳过门禁、以引擎 JSON 为准）由 `flow.yaml` 的步骤表 + `SKILL.md` 铁律统一规定，二者互补、不重复。

1. **交互询问与进度展示规则** — 仅调试启动阶段一次性采集全部参数写入配置；运行中仅固定阻塞场景触发 `🛑人工暂停`。每到达新步骤允许用一句话展示当前步骤；进入调试循环时还需简述循环次数与原因。上述内容仅为进度展示，不得展开分析、根因、日志或结论汇报。暂停场景细则见 `refs/pause-scenarios.md`。

2. **Git 版本管控规则** — 被调试项目全程禁止 `git push`，所有分支/提交/回退仅在本地操作；临时 CHESHI 代码与正式业务修复代码强制隔离。唯一例外是 Skill 报告发布：报告生成后必须通过 `scripts/git_sync.py` 将报告与 `data/debug-history.yaml` 提交并推送公司 GitLab 专用分支。详见 `refs/git-rules.md`。

3. **CHESHI 统一宏规范** — 全局仅使用 `CHESHI` 单一调试总开关（Bit 位掩码或数值分级）；所有新增调试输出、采集器和 Flush 路径必须受 `CHESHI` 包裹。通信层/ISR 仅采集快照，由 `main` 主循环输出；宏定义集中写在 `main.c` 头部，调试结束整段删除。详见 `refs/cheshi-macro.md`。

4. **动态路径加载规则** — 所有脚本运行时自动读取项目目录下的 `embedded-debug-config.json`；`uv4_path` 固定为 `C:\Keil_v5\UV4\UV4.exe` 自动填入，不询问用户。详见 `refs/config-format.md`。

5. **8 轮迭代上限** — 自动加打印满 8 轮仍无法定位故障时，必须触发人工求助。详见 `refs/debug-loop.md`。

6. **持续运行规则（运行中不可随意停下）** — 流程一旦进入执行，AI 必须持续驱动引擎按 `seq` 推进，**非必要情况一律不得停下/退出流程**：
   - **非必要情况（禁止停下）**：需要修改代码、需要向用户询问、需要等待确认、某一步暂未定位等——这些本就是流程内的正常环节（`edit_source` / `ask_user` / 迭代回退），必须在线内完成，不得借机中止整个流程或要求用户手动接管。
   - **必要情况（允许 `🛑人工暂停`）**：仅限真正阻断流程、AI 无力绕过的硬阻塞，例如：
     1. 程序卡死、目标板死机/看门狗失效，必须断电重启才能继续；
     2. 硬件缺失或不可用（无目标板、无下载器、串口/调试器丢失、供电异常），流程客观上无法推进。
   - 除以上必要情况外，AI 不得主动停下流程；即便需要用户补充信息，也应走流程内的 `ask_user` 步骤（如 seq 10 追问环节），而非终止运行。必要暂停场景细则见 `refs/pause-scenarios.md`。
