# /kzl 报告 — 根据当前对话生成详细报告并保存记忆

> 处理 `/kzl 报告`、`/kzl 生成报告`、`/kzl 输出报告`，可附带报告标题或额外要求。
> 此命令是独立快捷命令，不启动、不检查、不推进工作流引擎。

## 强制原则

1. 以当前可见对话为主线，结合本轮已读取的源码、Git 差异、终端命令输出和日志生成报告。
2. 使用 `templates/report.md` 的全部章节；内容必须详细、可追溯，不能只输出五项摘要。
3. 对话中没有证据的信息写“未提供”或“未验证”，禁止猜测、伪造日志、提交号和测试结果。
4. 编译成功不等于故障已解决；没有板端日志时必须明确验证边界。
5. 报告必须写入文件，同时更新项目历史索引和持久记忆；三项全部成功后才算命令完成。

## Step 1：确定报告工程

- 优先使用当前活动文件所属的已配置工程。
- 使用 `scripts/path_config.py` 解析并读取工作区配置，以 `projects` 数组为准。
- 无法唯一判断时，展示候选工程并询问用户选择。
- 配置不存在时仍可根据当前工作区生成对话报告，但必须注明工程配置未提供。

## Step 2：收集当前对话证据

整理用户诉求、故障现象、预期行为、复现条件、后续确认、源码分析、实际修改、编译下载、串口监听、终端输出、相关提交、失败尝试和未完成事项。只在必要时补读本轮涉及的文件、Git diff 或已知日志，不得为了填满模板虚构内容。

## Step 3：生成详细报告文件

- 模板：`templates/report.md`
- 位置：工作区 Skill 专属报告目录，目录由集中配置解析，文件名为
  `{YYYY-MM-DD}_{简短标题}.md`。
- 多项目共用工作区报告目录；报告正文必须列明涉及的项目名称和实际工程路径。
- 标题未提供时，根据对话自动生成不超过 30 个中文字符的文件名，并替换 Windows 非法字符。
- 同名文件已存在时追加 `_HHmmss`，禁止覆盖旧报告。
- 报告正文必须覆盖模板全部 11 节，最终回复给出绝对路径。

## Step 4：更新项目调试历史索引

将摘要追加到本 Skill 仓库的 `data/debug-history.yaml`：

```yaml
- date: "YYYY-MM-DD"
  desc: "简短标题"
  fault: "故障/任务摘要"
  root_cause: "根因；未确认则写未确认"
  status: "已解决/部分解决/未解决/仅记录"
  verification: "实际完成的验证摘要"
  report: "报告绝对路径"
  source: "conversation"
```

不得覆盖已有记录；同一报告路径已存在时不得重复追加。

## Step 5：保存持久记忆

1. 先查看 `/memories/`，优先更新已有 `embedded-debug-workflow.md`，避免创建重复文件。
2. 追加简洁条目：日期、工程、现象、根因、修复、验证状态、报告绝对路径。
3. 只保存可复用结论和报告索引，不复制整篇报告，不保存凭据、密钥或大段原始日志。
4. 若是当前仓库约定或工具行为，可同步写入 `/memories/repo/`；仅本次临时上下文不得写入持久用户记忆。

## Step 6：完成反馈

确认报告文件、`data/debug-history.yaml` 和记忆均已保存后，报告文件绝对路径、故障状态、一句话结论、索引与记忆保存状态，以及未验证事项。随后停止，不进入或推进调试流程。
